# TimescaleDB SCADA Lab

A runnable PostgreSQL + TimescaleDB portfolio: a generic solar + battery
SCADA platform with realistic synthetic telemetry, a living control queue,
a data-coherent alarm board, production-pattern policies (compression,
continuous aggregates, logged retention), and a DBA monitoring toolkit.

**One command, no database tools required — just Docker Desktop:**

```bash
git clone https://github.com/MesaDatabase/timescaledb-scada-lab.git
cd timescaledb-scada-lab
./demo.sh            # Windows: .\demo.ps1
```

~2 minutes later (first run downloads the image) you'll have built the
schema, generated ~800K readings across two solar sites, compressed 50+
chunks, materialized hourly/daily continuous aggregates, watched retention
drop aged chunks live, and scrolled a narrated tour: clipped solar curves,
storm-day energy dips, equipment outages with matching data gaps and alarms,
and a control queue with a full audit trail.

```bash
./demo.sh psql       # poke around yourself
./demo.sh monitor    # run the full DBA diagnostics suite
./demo.sh clean      # erase everything
```

Or connect your own tools: `psql -h localhost -p 5439 -U scada -d scada_demo`

## What's in here

| Folder | Contents |
|---|---|
| `01_monitoring/` | Production-safe diagnostics: sessions, locks, top queries (version-adaptive pg_stat_statements), hypertable/chunk sizing, background jobs, compression ratios |
| `02_schema/` | The platform: devices/points catalog, telemetry + control + alarm hypertables, replay-safe upserts, dual-write audit pattern, gapfill/rollup/energy analytics API |
| `03_data_generation/` | Config-driven synthetic plant: solar physics, weather, outages — all coherent, all inspectable, all tunable in `scada.demo_config` |
| `04_policies/` | Compression policies, real-time + hierarchical continuous aggregates, logged custom retention job — including a live downsampling demo |
| `05_performance/` | Reproducible blocking-lock walkthrough (benchmarks and EXPLAIN tours coming) |

Every script carries a standard header: Purpose, Safe in prod, Requires,
Compatibility, Notes. The design commentary lives in the SQL — read the
comments.

## Five things to look for

1. **Clipped solar curves** — clear-day inverter power flat-tops at the AC
   rating because the DC field is oversized. The generator explains the
   physics in its comments.
2. **Gaps, then gapfill** — outages cut telemetry mid-stream;
   `readings_gapfilled()` carries values through, flagging interpolation vs
   measurement.
3. **Audit ≠ queue** — every control request walks its lifecycle through one
   function that keeps the live queue current while history records every
   transition.
4. **Downsampling, live** — raw readings retain 45 days against 60 generated:
   retention visibly drops chunks while the continuous aggregates keep full
   history.
5. **An alarm board the charts agree with** — over-temperature alarms are
   generated by scanning the data for actual threshold crossings.

## How the one-button works

`docker-compose.yml` declares a single service — TimescaleDB (PostgreSQL 16)
— with the repo mounted read-only at `/repo` inside the container. `demo.sh`
starts it, polls `pg_isready` until the database answers, then runs every
SQL file in order **using the psql inside the container**, so the host
machine needs nothing but Docker. Stop with `down` (data persists in a
Docker volume) or `clean` (volume deleted, pristine reset).

## Requirements

- Docker Desktop (macOS / Windows / Linux). That's it.
- ~1 GB disk for image + default data volume.
